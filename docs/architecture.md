# 架构决策记录

本文档记录 Minesweeper Transformer 项目的关键架构决策。每个决策包含背景、考虑的方案、最终选择和理由。

## ADR-001: 概率蒸馏 vs 硬标签分类

- **状态**: ✅ 接受
- **日期**: 2026-05-27

### 背景
训练数据需要标签。传统做法是 BCE 二分类（是雷/不是雷）。但扫雷存在 NP-完全的约束一致性难题 — 某些局面的某些格子无法确定是否为雷。

### 考虑的方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: BCE 硬标签 | 0/1 标签，判断是否是雷 | 简单直接 | 丢失不确定性信息，模糊局面学习困难 |
| B: MSE 概率蒸馏 | 学习 ProbabilitySolver 的精确 P(mine) | 保留不确定性、支持模糊局面 | 需要求解器计算标签 |
| C: 混合 | 确定格用BCE，模糊格用MSE | 两全其美 | 工程复杂度高 |

### 决策
**采用方案 B（MSE 概率蒸馏）**。

理由:
1. 不确定性是扫雷的核心挑战（NP-完全），丢弃概率信息会让模型学习错误的确定性
2. 动作选择时选 P(mine) 最低格优于选"不是雷"格（因为可能没有确定的"不是雷"格）
3. 蒸馏 ProbabilitySolver 的概率分布比硬标签提供了更丰富的训练信号
4. 实际效果验证：概率蒸馏训练出的模型有更强的零样本泛化能力

## ADR-002: 通道设计 (11-ch)

- **状态**: ✅ 接受
- **日期**: 2026-05-27

### 背景
需要设计输入通道来表示棋盘状态。

### 决策
使用 11 通道：
- [0] covered: 1=未翻开, 0=已翻开
- [1] flagged: 1=已标雷, 0=未标
- [2:10] numbers 1-8: one-hot 编码翻开格数字
- [10] mines_remaining_ratio: 剩余雷数/总雷数（提供全局约束）

理由:
1. One-hot 数字通道让 CNN 可以独立学习每种数字的邻域模式
2. mines_remaining_ratio 提供全局信息（类似棋类中的"剩余步数"通道），帮助模型理解全局约束
3. 11 通道比纯位图更结构化，比全 one-hot 标记更紧凑

## ADR-003: 模型架构 (CNN + Transformer 混合)

- **状态**: ✅ 接受
- **日期**: 2026-05-27

### 背景
扫雷同时需要局部模式识别（CNN 擅长的数字-邻域关系）和全局约束推理（Transformer 擅长的远距离依赖）。

### 决策
CNN (3层 3×3 Conv, 64ch) → InterpolatablePE → Transformer (4层, d=64, 4头) → 2ch 输出。

理由:
1. CNN 提取局部邻域特征（"这个数字意味着周围有几个雷"）
2. Transformer 推理全局约束（"剩余雷数要求至少还有一个雷在这片区域"）
3. InterpolatablePE 让模型可以泛化到不同棋盘尺寸
4. 297K 参数 — 足够小，训练快，不易过拟合

### 已知局限
- 全局注意力在 16×16 以上棋盘可能成为瓶颈 → 未来考虑局部窗口注意力
- 当前未使用 prev_probs 通道（设计预留，refinement 训练时启用）

## ADR-004: 迭代 Refinement

- **状态**: ✅ 接受
- **日期**: 2026-05-28

### 背景
单次前向推理可能产生不一致的预测（例如预测某个区域有 2 个雷，但数字只允许 1 个）。迭代 refinement 让模型反复修正自己的输出。

### 决策
- 全局策略: `src/config/training_policy.py` 是 refine 参数唯一默认来源
- 训练: 每个 batch 随机采样 refinement 步数 `k ∈ [1, 16]`，只对第 k 步计算 loss
- 迭代间使用 `detach()` 切断 BPTT，避免显存和梯度爆炸
- 推理/评估: `max_refine_steps=16`，使用收敛检测 `max|P_t - P_{t-1}| < 1e-3` 早停
- RL: rollout 与梯度重算统一使用 16 步 refinement，保证 on-policy 一致性
- 输入: 初始 `prev_probs=0.5`，后续迭代使用上一轮输出

理由:
1. 随机 k 让模型对不同推理深度都可用，避免只适配固定步数
2. detach() 防止 BPTT 爆炸（类似 truncated BPTT），大幅降低显存
3. confidence head 没有可靠的显式停止监督，收敛检测比 confidence 阈值更稳定
4. 10×10/40 高密度评估需要更高推理上限，16 步给收敛留余量
5. CLI 不再暴露 refine 参数，避免监督训练、RL、评估出现局部默认值不一致

## ADR-005: 密度课程 vs 混合数据

- **状态**: ✅ 接受（两条路线并存）
- **日期**: 2026-05-28

### 背景
如何组织训练数据？逐步提升难度（课程学习）还是混合所有难度？

### 决策
两条路线并存，互不冲突:
1. **分阶段课程** (`train_stage.py`): S1(8×8/10) → S2(8×8/20) → S3(10×10/40)，每阶段继承权重
2. **混合数据** (`generate_data.py --mixed`): 一次生成可变尺寸+密度，单次训练

当前主力使用分阶段路线（更可控，每阶段验证）。

## ADR-006: RL 算法选择 (REINFORCE vs PPO)

- **状态**: ✅ 接受
- **日期**: 2026-05-30（更新 2026-05-31）

### 背景
监督学习后需要 RL 微调提升实战能力。

### 决策
**采用 REINFORCE**。

理由:
1. 从监督权重 warm start，策略已经很好了，不需要 PPO 的稳定性保证
2. 扫雷每局 20-80 步，轨迹短，REINFORCE 的高方差问题不严重
3. 工程简单，更容易调试

### RL 环境设计决策

#### mine_continue: 训练 True, 评估 False
- **训练时 `mine_continue=True`**: 模型踩雷后自动标雷，游戏继续。让模型从错误中学习而不提前终止
- **评估时 `mine_continue=False`**: 踩雷即结束，反映真实扫雷规则下的胜率
- **理由**: 分离训练信号（密集反馈）和评估标准（真实规则）

#### 奖励设计
```
reveal_safe      = +1.0
floodfill_bonus  = +0.05 × max(0, cells_revealed - 1)
hit_mine         = -20.0
win_reward       = none
```
- **即时奖励**: 扫雷动作没有“牺牲当前换未来”的规划收益，训练信号只绑定当前动作结果
- **小权重 floodfill**: 保留快速结束游戏的倾向，但避免模型为追求大 floodfill 奖励而赌博
- **无 win bonus**: 避免最后一步获得与动作无关的巨额梯度
- **无预揭开补分**: 开局已翻格属于环境状态，不归因到模型最后一步动作

#### 固定尺寸 vs 混合
- **focused RL (10×10/40)**: 单一尺寸/密度，训练信号聚焦
- **mixed_env 仅实验使用**: 训练入口默认固定棋盘，显式 `--mixed` 才启用混合池
- **理由**: 主目标是把 S3 checkpoint 在目标分布上推近 100%，固定分布更容易诊断

#### 棋盘池
- `generate_rl_pool.py` 负责预生成 self-validated board pool
- `train_rl.py` 训练侧只读取 pool，不在训练过程中构造或补齐数据
- `first_done=True`: 模型从 after-first-click 态出发（与训练数据一致），避免 OOD 全覆棋盘

#### 提前退出：收敛检测
- 推理时 `max|P_t - P_{t-1}| < 1e-3` → 停止
- 替代置信度方案（confidence head 无显式监督，无法可靠达到 0.95）
- 实测 S3 模型 90% 样本在 5 步内收敛（max 16）

#### Advantage
- 当前实现使用 batch 内即时奖励标准化降低方差
- `baseline` 仅作为训练日志诊断指标，不参与 advantage 计算

## ADR-007: 三阶段预训练路线

- **状态**: ✅ 接受
- **日期**: 2026-05-30

### 背景
原来有 S1-S4 分阶段 + mixed 混合路线并存，各自独立运行，造成混淆（Agent 曾用旧 S1 训练后才发现不是最新方案）。

### 决策
统一为三阶段监督预训练课程：
1. **S1 (规则)**: 8×8/10雷, 2 epochs, 学习基本扫雷规则
2. **S2 (密度)**: 8×8/20雷, 2 epochs, 继承 S1 → 学习雷密度可变
3. **S3 (高密度泛化)**: 8×8/25雷, 5 epochs, 继承 S2 → 密度对齐 10×10/40

理由:
1. “先教规则，再教密度，再对齐目标密度”的课程设计可控且易复现
2. S3 的 8×8/25 密度约 39%，接近目标 10×10/40 的 40%
3. 统一入口 `train_stage.py --stage S1/S2/S3`，消除“该跑哪个阶段”的困惑
4. 评估目标明确: 10×10/40雷 零样本，然后用 RL 在目标分布保守微调
5. 旧 `S1.5/S2.5/S2.75/S3L/S4L` 保留为 `--legacy_stage`，不属于默认主线

## ADR-003: V3/V4 双轨重构 (Dual-Track Refinement)

- **状态**: ✅ 接受
- **日期**: 2026-06-02

### 背景
模型在处理长程逻辑推导（如反证法）时，原有的单通道概率传递（`prev_probs`）缺乏足够的信息容量来作为"草稿本"。为了让模型能够进行深度思考，引入了隐式记忆（`mem_state`）。

### 考虑的方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: CNN 移出循环 | CNN只在初始阶段提取特征，Transformer循环处理 `mem_state` | 计算量极小 | 丧失了局部空间归纳偏置，收敛极差 |
| B: 双轨推演架构 | CNN 每次都处理 `[board, prev_probs]`，Transformer 处理 `mem_state` + CNN特征 | 保留空间偏置，且赋予高维记忆能力 | 参数量略增，计算量较大 |

### 决策
**采用方案 B（双轨推演架构）**。

理由:
1. 扫雷问题极度依赖局部的空间特征，必须将 CNN 保留在 Refine 循环中，否则模型连基础规则都难以学会。
2. 引入 64 维 `mem_state` 作为 Transformer 内部的隐式记忆，允许模型保留逻辑推演的中间状态。
3. 增加 `DecoderHead`（1x1 Conv, `bias=True`）将高维记忆转化为最终概率，其中 `bias=True` 为高密度棋盘提供了雷概率的全局先验。

## ADR-004: 废弃 PonderNet 与恢复固定步长 BPTT

- **状态**: ✅ 接受
- **日期**: 2026-06-02

### 背景
曾尝试使用 PonderNet（引入 Halt Head 和 Ponder Penalty）让模型自适应决定思考步数，但发现这导致了梯度冲突和 Loss 剧烈震荡，模型甚至变得"懒惰"提前退出。

### 决策
**废弃 PonderNet，回归固定步长 BPTT (Backpropagation Through Time)**。

理由:
1. 在训练阶段采用固定较短步长（`train_max_steps=4`）进行完整的 BPTT，确保梯度能穿透足够深度的推演逻辑，且避免 Ponder 带来的优化不稳定。
2. 推理阶段放开步数限制（`eval_max_steps=16`），通过前后两次输出的物理概率差值（ΔP）作为收敛判据来物理早停。
3. 训练与推理的步数解耦，极大提升了训练速度，同时保持了推理时的深度思考能力。
