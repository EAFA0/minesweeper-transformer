# Bug Report: S3 评估阶段胜率崩塌 (S3 Evaluation Collapse)

## 现象描述 (Symptoms)
在应用了最新的 V3/V4 双轨架构（Dual-Track Architecture, CNN + Transformer with `mem_state`）后，模型在 S1 (8x8/10雷) 和 S2 (8x8/20雷) 阶段表现极佳（S1 胜率 98.5%）。然而，在 S3 (8x8/32雷，50% 雷密度) 阶段出现了严重的指标矛盾：
- **训练阶段 (Train)**: 完美收敛，Val Loss 低至 0.0012，Val Acc 达到 98.6%，Action Acc 99.7%。
- **评估阶段 (Evaluate)**: 实际游玩时，胜率暴跌至约 36.7%，尽管在动作选择准确率 (Action Acc) 上仍有 95.9%。

## 环境与上下文 (Context)
- **数据源**: 使用 `scripts/train_stage.py` 和 `src/data/generator.py` 生成的**无猜扫雷 (No-Guess)** 数据集。数据集保证所有的盘面在逻辑上都是可解的。
- **架构**: 
  - CNN 处理 `[board, prev_probs]` 提取局部空间特征。
  - Transformer 处理 CNN 特征并维护 `mem_state`（64维隐式记忆）。
  - 训练时采用固定 4 步 BPTT 展开。
  - 推理时最多允许 16 步迭代，通过概率差值阈值进行物理早停。
- **状态管理**: 训练与评估时，每次输入新盘面进行推演，初始的 `prev_probs` 和 `mem_state` 均严格初始化为全 0。不存在评估时状态跨步遗留的问题。

## 已排除的错误假设 (Discarded Hypotheses)
1. **评估时 `mem_state` 跨步未清零/未继承**：
   - 之前曾猜测评估时需要继承上一回合的 `mem_state`。但用户已指出，训练数据是提供完整的盘面进行**单步静态推演**（给定盘面，推断未翻开格子的概率），每次推演的 `mem_state` 都是从零开始的。评估时也是每一步给定当前可见盘面从零推演。两者的输入输出模型完全一致。
2. **训练数据由于过滤了"猜测"导致畸变**：
   - 曾猜测 50% 密度的棋盘如果强制过滤掉需要猜测的局，会导致生成的棋盘过于特殊（OOD）。但用户强调，这就是"无猜扫雷"，训练和评估的 Case 都是同源的无猜局面。如果是同源数据，模型在验证集上 Acc 高达 98.6%，在评估集上就不应该只有 36% 胜率。

## 潜在排查方向建议 (Next Steps for Reviewing Agent)
1. **评估脚本 (evaluate.py) 与训练数据分布的微小偏差**:
   - 是否存在某种实现细节，导致 `evaluate.py` 实际遇到的盘面序列，与 `dataset.py` 生成的盘面在统计学特征上存在断层？例如：第一步点击的安全性、评估过程中的展开逻辑等。
2. **自回归曝光偏差 (Exposure Bias)**:
   - 训练时是基于完全正确的历史状态预测下一步（Teacher Forcing）。而在评估时，虽然每步都重新计算，但一旦模型某一步预测出了微小的偏差并据此翻开了一个格子（或者选择了错误的概率极小值），这个微小偏差的累积可能将模型带入了一个从未在训练集中见过的局面。
3. **Action Acc 95.9% 与 Win Rate 36.7% 的关系**:
   - 95.9% 的动作准确率看起来很高，但在扫雷中，一步走错即全盘皆输。如果一局需要走 20 步，0.959^20 ≈ 43%，这与 36.7% 的胜率在数学期望上是吻合的。这说明在 8x8/32 这种高难度下，模型并非完全失效，而是容错率为零放大了 4% 的误差。是否需要调整 Loss 权重（例如更重的错杀惩罚），或者在评估时引入搜索树（MCTS）？

## 结论 (Conclusion)
当前代码库已被清理并提交（Commit: `fix: S3 evaluation mismatch review prep` 等）。原开发 Agent 已因排查方向偏离被用户中止。请接手的 Agent 从代码的**训练数据生成**与**评估脚本实际运行机制**之间的差异入手，或者重新审视扫雷任务在高密度下的极低容错率特性。
